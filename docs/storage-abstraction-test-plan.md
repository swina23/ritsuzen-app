# 保存層の抽象化 実機確認項目（v1.10.0）

対象: `docs/storage-abstraction-design.md` の実装。`cloud` ブランチ。

コードレビューでは埋められない部分だけを挙げる。**A〜C は必須**、D 以降は余力に応じて。

## 実施結果（2026-07-23 / localhost:5173 / Brave）

| 節 | 結果 |
|---|---|
| A | A-2 以外すべて合格（A-10 を含む） |
| B | B-2〜B-6 すべて合格 |
| C | **C-1〜C-10 すべて合格** |
| D | D-1〜D-4 合格。D-5 未実施 |
| E | E-1〜E-5 + E-4b すべて合格 |
| F・G・H | 合格 |
| I | I-1 合格、I-2 は「残留する」ことを実測（既知の挙動・スコープ外） |

**未実施の2件はどちらも空データ状態が前提**で、既存の記録を消さないと作れないため見送った。

- **A-2**（起動時に「データを読み込み中…」が出ない）— ローカルは初回スナップショットを同期的に流す設計。実際にちらつきは観測されなかったが、既存データ有りの状態での観察なので「合格」とは断定していない
- **D-5**（記録0件で通算成績タブ）— 空データ状態が必要

**この確認中に見つけて直した不具合**: `StatusMessage` のクラス名（下の「実機確認で判明した事項」を参照）。本改修とは無関係の既存バグ。

**持ち越し**

- 共用端末でログアウトしても Firestore のキャッシュがディスクに残る（I-2）。アプリ経由の漏洩経路は無いので今回は直さない
- **Firestore SDK が未ログインユーザーにもダウンロードされる。** `src/lib/firebase.ts` が `firebase/firestore` を静的 import し、それを `AuthContext` 経由で常時マウントの `AuthProvider` が引き込むため、`FirestoreBackend` の動的 import で切り出せているのは 1.67 kB のラッパーだけで、SDK 本体はメインバンドル（約2MB）に残っている。動作の不具合ではなくバンドルサイズの問題。直すには `firebase.ts` を auth 用と firestore 用に分割する必要があり、Firestore の初期化タイミングが変わるため C 節と H 節の再確認が要る。本改修とは分けて対応する

## 準備

```bash
npm run dev
```

DevTools を開き、次の場所を見られるようにしておく。

## A. 端末保存モード（未ログイン）の基本動作 ★必須

前提: ログアウト状態。`ritsuzen-app-local-v2` も `ritsuzen-app-data` も無い状態から始める（両方削除してリロード）。

## B. 旧 `main`（localStorage 版）からの移行 ★必須

**移行は端末保存モードでのみ走る。** `ritsuzen-app-local-v2` が存在しないときだけ `ritsuzen-app-data` を読む片道移行で、旧キーは削除しない。

| # | 手順 | 期待 |
|---|---|---|
| B-1 | `ritsuzen-app-local-v2` を削除し、旧版で作ったデータを `ritsuzen-app-data` に入れる（旧 `main` を実際に使うか、旧形式のJSONを手で入れる） | — |
| B-2 | 未ログインでリロード | 旧データの大会・参加者マスターが全部見える |
| B-3 | 旧版で**進行中だった**大会（旧形式では `currentCompetition` として配列の外にあった）| 「現在の大会」として復元される。履歴側に重複して現れない |
| B-4 | `ritsuzen-app-data` を確認 | **消えずに残っている**（移行に失敗したとき戻れるようにするため。意図的な仕様） |
| B-5 | 何か記録を入力してリロード | 以後は `ritsuzen-app-local-v2` だけが更新され、`ritsuzen-app-data` は据え置き |
| B-6 | `ritsuzen-app-data` に壊れたJSON（例: `{`）を入れ、v2キーを消してリロード | アプリは落ちない。Console に警告が出て、空の状態で起動する |

```json
  "competitions": [],

```

## C. バックエンドの切り替え ★必須・最重要

**今回の改修で最も危険な箇所**。`AuthGate` を撤去したことで `CompetitionContext` が切り替えをまたいで生き残るようになったため、ローカルのデータがクラウドへ（またはその逆に）混入しないかを見る。

判定の要は「**切り替え後の画面に、切り替え前のデータが1件でも残っていないか**」。

| # | 手順 | 期待 |
|---|---|---|
| C-1 | 端末保存モードで大会を作り、参加者と記録を入れておく（＝ローカル側に固有のデータを作る） | — |
| C-2 | ヘッダーの「クラウド保存を使う」→ Googleログイン | ログイン画面がダイアログとして重なって開き、成功すると自動で閉じる |
| C-3 | ログイン直後の画面 | バッジが `☁️ クラウド保存` に変わる。**C-1 で入れたローカルの大会が消え、クラウド側のデータに入れ替わる**（自動では引き継がれない。仕様どおり） |
| C-4 | `ritsuzen-app-local-v2` の中身 | **C-1 のまま変化していない**。クラウドの大会がローカルに書き込まれていないこと |
| C-5 | Firestore コンソール（または再ログイン後の表示） | **C-1 のローカルの大会がクラウドに作られていない**こと。ここが一番の見どころ |
| C-6 | ログアウト | バッジが `📱 この端末に保存` に戻り、C-1 のローカルデータが復帰する。クラウドの大会は見えない |
| C-7 | C-2〜C-6 を数回繰り返す | 毎回同じ。データが混ざったり、片方が消えたりしない |
| C-8 | 記録入力画面で的中を入力している最中にログアウト | 画面が壊れない。Console に例外が出ない。最後の入力がクラウドに書かれる（ローカルには書かれない） |
| C-9 | 許可リストに無いGoogleアカウントでログイン | 「クラウド保存の利用が許可されていません」と出る。**閉じればそのまま端末保存で使い続けられる**（記録は消えない）。ヘッダーは `⚠️ 未登録のアカウント` |

### C-10 圏外をまたぐ切り替え（今回の修正点）

レビューで見つけた「1段目の書き込みを待つ間にバックエンドが切り替わる」経路の確認。

1. クラウドモードで大会を開き、DevTools の Network を **Offline** にする
2. 記録を数本入力する（同期は完了しない）
3. その状態でログアウトする
4. Network を **Online** に戻す

期待: エラートーストが連発しない。ログアウト後のローカル側のデータが書き換わらない。オンラインに戻したとき、2 で入れた記録がクラウド側に着地している（次回ログイン時に見える）。

## F. 容量オーバー（端末保存モードのみ）

localStorage は 5MB 程度で頭打ちになる。無料モードの実質的な上限なので、そこで何が起きるかを見ておく。

1. Console で意図的に容量を埋める:
   ```js
   localStorage.setItem('__filler__', 'x'.repeat(4_500_000))
   ```
2. アプリで記録を入力する

期待: 「この端末の保存容量が上限に達しました。古い大会を削除するか、クラウド保存をご検討ください。」が画面に出る。アプリは落ちず、入力中の記録が画面上から消えない（メモリ上の状態は意図的に巻き戻していない）。

確認後 `localStorage.removeItem('__filler__')` で戻す。

## I. 共用端末での残留（既存の懸念、本改修とは独立）

設計書 §11 が「実機で確認する」としている項目。Firestore の IndexedDB 永続キャッシュは `dispose()` では消えない。

> **当初の文面は誤りだったので書き換えた。** 元は「許可済みアカウントAでログイン → ログアウト → 許可済みアカウントBでログイン。期待: BにAのデータが見えない」としていたが、`firestore.rules` はメンバー間で区画を分けておらず（`match /{document=**}` に許可リストの2アカウントが揃って read/write 可）、**AとBは同一のデータを共有するのが仕様**。BにAのデータが見えるのは正しい動作で、これを不具合として観測することはできない。実際に確かめる価値があるのは次の2点。

| # | 手順 | 期待 |
|---|---|---|
| I-1 | 許可リストに無いアカウントでログインした状態で各タブを見る | クラウドの大会・マスターが**画面に一切出ない**。`CompetitionContext.tsx` は `status === 'signedIn'` のときだけ `FirestoreBackend` を動的importし、`unauthorized` は `LocalStorageBackend` に落ちるため（C-9 と同時に確認できる） |
| I-2 | 許可済みアカウントでログイン → キャッシュ件数を控える → ログアウト → もう一度数える | 件数が**減らない**（＝残留する）。これは既知の挙動で、アプリの不具合ではない |

**I-2 実測（2026-07-23、localhost:5173）**

`indexedDB` の `firestore/[DEFAULT]/ritsuzen-app/main` を直接数えた結果。

| ストア | ログイン中 | ログアウト後 |
|---|---|---|
| `remoteDocumentsV14` | 42 | 42 |
| `targetDocuments` | 81 | 81 |
| `targets` | 3 | 3 |

1件も減らない。中身は Firestore の生ドキュメントのままで、`competitions/comp-...` の記録が未ログイン状態の DevTools から素で読める。

一方で、消えるものは正しく消えている。

- `firebaseLocalStorageDb`（認証トークン）は **0件**になる
- ヘッダーは「クラウド保存を使う」に戻り、画面に出るのは端末保存の大会だけ

**結論**: アプリの画面経由でデータが漏れる経路は無い。残るのは「共用端末でログアウトしても、DevTools を開ける人はチームの記録を読める」というディスク上の残留のみ。今回のスコープ外だが、共用端末での運用を想定するならログアウト時の `clearIndexedDbPersistence()` 追加を別途検討する。

## 実機確認で判明した事項（本改修とは独立）

- **広告ブロッカーがクラウド保存を止める。** Brave Shields を有効にした状態だと `firestore.googleapis.com` への POST が `ERR_BLOCKED_BY_CLIENT` で落ちる。Shields を Down にすると消えるので、アプリ側の不具合ではない。ただし**有料の売りであるクラウド保存が、ブロッカー利用者の環境で丸ごと動かない**可能性があるため、本番ドメインでも同様かを別途確認する。
- **A-10 で既存バグを1件発見・修正した。** `StatusMessage` が `className="import-status"` を使っていたが CSS には `.status-message` しか無く、当たるルールが無いため既定の白文字のまま白いカードに表示され、インポート結果のメッセージが絵文字しか見えなかった。クラス名を `status-message` に修正済み。本改修とは無関係の既存不具合。
- **COOP の警告は無視してよい。** ログイン時に `Cross-Origin-Opener-Policy policy would block the window.closed call.` が多数出るが、Firebase の `signInWithPopup` がポップアップの開閉を監視する際の既知の挙動。ログインは成功する。改修前から出ている。
